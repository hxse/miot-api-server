# 阶段计划

## Stage 1：冻结认证刷新真值

- 建立当前 task。
- 明确 curl 业务接口与网页登录入口的认证刷新差异。
- 冻结 Provider 业务入口刷新 contract。

验收条件：

- `00_meta.md`、`01_context`、`02_spec` 已覆盖任务级别、范围、方案取舍、正式 contract、失败语义、语义新增与冻结清单。

## Stage 2：Provider 刷新入口修复

- 在 `MijiaProvider` 中收口业务接口获取认证 API 的逻辑。
- 业务入口在认证文件存在但不可用时尝试一次底层刷新。
- 认证刷新请求必须显式设置 timeout。
- 刷新成功后继续复用刷新后的 API 实例。
- 刷新失败后返回现有 `authentication_required` 语义。

验收条件：

- curl 业务接口路径不再被 `api.available=false` 提前阻断刷新。
- 登录页扫码流程不发生语义漂移。

## Stage 3：验证与最终审阅

- 执行静态/构建类验证。
- 执行离线控制流验证，覆盖可用、可刷新、不可刷新、无认证文件、损坏认证文件、缺字段认证文件分支。
- 回填执行记录与最终验收。

验收条件：

- `04_review` 已记录验证结果、残余风险与最终结论。
