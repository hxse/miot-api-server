# 方案取舍

## 方案 1：只在网页登录页刷新

保持当前实现不变，把“curl 失败后回网页登录”写成使用说明。

该方案实现成本最低，但它把自动化调用的可用性依赖到手工网页登录动作上。用户使用 curl、Tasker 或其他自动化客户端时，业务入口无法自行利用仍可刷新的旧认证文件，等同于把底层刷新能力藏在网页登录页里。

本任务不选择该方案。

## 方案 2：业务接口入口先刷新再继续

Provider 在业务入口需要米家 API 实例时，先尝试让底层 `mijiaAPI` 完成认证刷新。刷新成功后保存认证文件并继续原业务请求；刷新失败后再返回 `authentication_required`。

该方案符合现有 Provider 边界，能统一 curl 与网页登录路径的认证语义，也不会改变对外控制接口形态。缺点是业务请求在旧认证文件失效时可能多一次米家账号服务请求，单次请求耗时会增加。

本任务选择该方案。

## 方案 3：新增后台定时刷新任务

服务启动后定期刷新米家登录态，让业务请求尽量不遇到认证失效。

该方案会引入新的后台生命周期、失败重试、并发互斥与可观测性要求。当前问题可以在业务入口同步解决，不需要新增长期运行的刷新调度器。

本任务不选择该方案。

## 方案 4：手写小米认证刷新协议

绕过 `mijiaAPI` 私有方法，重新实现小米账号 `passToken` 到 `serviceToken` 的刷新链路。

该方案会产生第二套认证实现，破坏当前“Provider 复用 `mijiaAPI==3.0.5`”的验证基线。除非后续任务明确升级认证底座，否则不应在本轮引入。

本任务不选择该方案。
