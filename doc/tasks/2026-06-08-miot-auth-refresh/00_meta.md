# 任务描述

本任务用于修复米家登录态刷新链路不一致的问题：自动化调用通过 curl 访问业务接口时，旧认证文件在可刷新状态下不应先被业务入口判定为不可用；网页登录入口可以刷新认证文件的能力需要收口到 Provider 的正式认证入口中。

# 任务级别

本任务定级为 `A` 类。原因是它调整米家认证状态流、业务接口失败语义与自动化调用可用性，影响 `/api/devices`、`/api/devices/{did}` 与 `/api/devices/{did}/power/on|off` 的核心前置条件；该链路属于高耦合、quietly wrong 风险较高的第三方云登录态处理。

# 任务范围

## In Scope

- 修复业务接口入口在米家认证文件可刷新时直接返回 `authentication_required` 的问题。
- 统一 curl/API 调用与网页登录入口对旧米家认证文件的刷新语义。
- 保持 `APP_TOKEN` Bearer 鉴权语义不变。
- 保持扫码登录、登录会话、认证文件路径与 Docker volume 持久化语义不变。
- 为刷新成功、刷新失败、无认证文件、损坏认证文件与缺字段认证文件关键分支补充离线验证。
- 更新 task index，并在任务验收中记录执行结果、验证缺口与残余风险。

## Out of Scope

- 不改造 `APP_TOKEN` 的生成、存储、轮换或浏览器记忆策略。
- 不升级或替换 `mijiaAPI==3.0.5`。
- 不新增第二套米家认证实现，不手写完整小米登录协议。
- 不扩展新的设备能力、通用 MIoT DSL 或控制接口。
- 不把真实米家扫码登录、真实设备云控、VPS 反向代理和外网连通性纳入本轮验证。
