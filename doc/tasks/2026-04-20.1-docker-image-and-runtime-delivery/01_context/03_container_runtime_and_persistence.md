# 容器运行时与持久化思路

## 为什么容器内必须有统一数据目录

当前应用至少有两类需要跨重启保留的状态：

- 米家认证文件。
- MIoT spec 缓存目录。

如果不把它们统一收口到容器内稳定路径，部署者就需要自己理解多组环境变量和多组挂载点，容器运行时也会更容易出现“镜像能启动，但登录状态和缓存全丢失”的伪成功状态。

因此，本任务采用单一运行时数据根目录：

- 容器内统一使用 `/data/mijia/` 承接认证文件与 spec 缓存。
- Docker 路线统一通过 volume 挂载这个目录，而不是让不同入口一部分走 bind mount、一部分走匿名 volume。

## 为什么容器内默认要监听 `0.0.0.0`

本地源码路径下，只有 `just uv-run` 被允许为了联调方便显式绑定 `0.0.0.0`，其他入口默认保持 `127.0.0.1`。这是因为宿主机启动会直接接触局域网和公网暴露面。

容器运行时不同：

- 容器内部绑定 `0.0.0.0` 只是为了让 Docker 端口映射能够工作。
- 真正是否对外暴露，仍由 `docker run -p`、宿主机防火墙与后续反向代理决定。

因此，容器内默认监听 `0.0.0.0` 是合理的运行时 contract，而不是对之前安全原则的反向推翻。

## 为什么本任务只冻结单镜像与 `docker run`

当前最需要的是一个明确、可复制、可在 VPS 上直接跑起来的交付路径：

- `docker build`
- `docker run`

`compose`、反向代理、进程管理、镜像仓库推送和 CI/CD 都属于更上层的部署编排。过早把这些一起塞进来，只会把当前 task 从“封装镜像”膨胀成“完整运维体系设计”。

## 为什么当前要统一走 volume 持久化

之前把 `just docker-run` 和“直接跑镜像”区分成不同持久化策略，会让行为变得不直观：

- 一条路径保存在宿主机目录。
- 另一条路径保存在 Docker 自己管理的卷里。

这会导致“缓存为什么还在”这种问题很难一眼说清。

因此，本任务收口为：

- Docker 路线统一视 `/data/mijia` 为 volume 挂载点。
- `just docker-run` 默认使用命名 volume，便于稳定复用缓存。
- 镜像自身也声明 `/data/mijia` 为 volume，避免把状态误认为镜像层内容。
- 官方推荐路径只保留一条固定的 `docker build` 和一条固定的 `docker run`，不再为镜像名、容器名、端口和 volume 暴露第二层配置入口。
