# 部署与安全边界

## Docker 化后哪些安全口径不变

这个 task 只改变交付方式，不改变当前应用内安全真值：

- `APP_TOKEN` 仍然必须显式提供。
- 服务仍然不得生成默认 token。
- 业务接口仍然统一使用 `Authorization: Bearer <TOKEN>`。
- 文档页与登录页的鉴权口径不放宽。

## Docker 化后新增的安全问题是什么

镜像交付会引入三类新的风险：

- 把敏感配置烤进镜像层，例如把 token 或认证文件写进镜像。
- 把容器内部监听地址、宿主机端口映射和公网暴露面混为一谈。
- 把临时容器当成可持久化运行环境，导致登录状态和缓存 silently 丢失。

## 当前任务的安全收口方式

本任务只冻结最小边界：

- token 只能通过运行时环境变量注入，不写进镜像。
- 认证文件和 spec 缓存必须通过挂载到容器内稳定数据目录实现持久化。
- 容器内部监听 `0.0.0.0`，但对外暴露仍必须由显式端口映射触发。
- HTTPS、Nginx、域名与公网防火墙仍留给后续部署任务。
