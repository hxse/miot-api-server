# 问题背景

当前仓库已经完成第一阶段应用闭环：

- `uv run` 启动路径可用。
- `APP_TOKEN` 失败语义已冻结。
- `/login`、`/docs`、`/redoc`、设备列表与 `power` 控制路径已落地。
- 认证文件与 spec 缓存已经有明确的本地持久化约束。

但部署层仍停留在“源码树 + 本机 Python 环境”的运行模型。这会带来三个直接问题：

- 本地和 VPS 的运行环境复用性差，部署前提仍然依赖宿主机 Python / `uv` / 依赖缓存状态。
- 当前应用虽然能跑，但还没有一个明确的“交付物”可以被复制到另一台机器后直接运行。
- 部署层最小安全边界还没有收口到容器 contract，例如 token 如何注入、持久化目录如何挂载、镜像内是否允许带默认配置、容器端口如何暴露。

因此，这个 task 的本质不是“换一种启动姿势”，而是把当前应用收口成稳定的容器交付物，并把部署层最小真值写成 task spec。
