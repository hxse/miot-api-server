# 方案对比与取舍

## 方案一：镜像内继续使用 `uv run --no-project` 从源码树启动

优点：

- 和当前本地 `just uv-run` 路线最接近。
- 改动看起来最小。

问题：

- 运行时依赖源码树布局、`uv` 的项目解析行为和镜像内工具链完整性。
- 容器启动时继续保留“开发入口”语义，会把本地调试约束和生产镜像约束混在一起。
- 更难解释“为什么 Docker 入口和本地调试入口本质一致但实现不一致”。

## 方案二：镜像构建期安装项目与依赖，运行时直接执行 `uvicorn`

优点：

- 镜像成为单一交付物，不依赖宿主机源码挂载。
- 容器运行期语义直接收口到“已安装应用 + 明确入口命令”。
- 更容易稳定约束镜像大小、启动方式与失败语义。

代价：

- Dockerfile 里需要显式处理依赖安装与应用安装。
- 需要明确容器内数据目录与默认运行参数。

## 方案三：引入自定义 shell entrypoint 脚本承接大量前置逻辑

优点：

- 容器启动前可以做更多检查和动态拼接。

问题：

- 容易把配置校验、目录创建、默认参数、环境变量回退都塞进第二套脚本逻辑。
- 会形成“应用内一套真值、entrypoint 脚本另一套真值”的隐性双轨。

## 当前收口

本任务选择方案二：

- 构建期完成依赖安装与应用安装。
- 运行期直接执行单一 `uvicorn` 入口。
- 只保留必要的极少量启动前置动作，例如确保持久化目录存在。

这样既能和当前应用骨架保持一致，又不会把本地 `uv run` 的开发入口硬塞进容器运行期。
