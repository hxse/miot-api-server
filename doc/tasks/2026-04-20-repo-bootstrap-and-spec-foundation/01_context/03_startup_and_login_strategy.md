# 登录与启动策略

## 为什么第一阶段必须包含登录能力

如果第一阶段只保留设备控制接口与 `GET /healthz`，却没有扫码登录能力，那么服务无法建立米家帐号会话，也无法稳定获得目标设备上下文。这不是“后续优化项”，而是控制链路的起点。

因此，第一阶段的最小闭环应至少覆盖：

- 米家帐号初始化登录。
- 一个不依赖 `curl` 的专用登录页面。
- 登录后获取设备列表。
- 直接按设备发起能力控制，而不是先维护一个全局绑定目标。
- 对外提供健康检查。

## 为什么不接受手工填写 `did/siid/piid`

手工填写 `did/siid/piid` 的问题不只是麻烦，而是把内部实现细节暴露给部署者：

- `did` 可以通过设备列表发现，不应要求用户先去抓内部标识再回填。
- `siid/piid` 属于 MIoT 规格细节，应该由 Provider 层自行解析或封装，不应成为第一阶段部署前置条件。
- 当前项目定位是轻量服务，不是底层协议调试工具。

因此，本任务把“用户无需手工配置 `siid/piid`”冻结为正式方向。后续实现允许内部解析和缓存这些结果，但不允许把这组字段作为部署者的主要输入成本。

## 为什么当前任务只先打通 `uv run`

当前仓库还没有 Dockerfile、镜像构建链路和完整运行目录结构。如果在这一轮同时把 Docker 路线写死，风险是把尚未冻结的容器细节误写成正式真值。

因此本任务先打通 `uv run`，原因有三点：

- 它是当前最短的本地调试闭环，能尽快验证应用入口、配置校验和最小安全约束。
- 后续无论进入本地容器调试还是 VPS 部署，都应该建立在同一个应用骨架之上。
- 先把最小应用与最小安全基线跑通，再进入容器化，会比同时并行冻结两套入口更稳。

## `uv run` 当前必须满足的语义

- 只运行一个统一的 FastAPI 应用入口。
- 当前通过 `uv run --no-project` 从源码树直接启动，不把项目打包构建作为本任务的运行前置。
- 必须显式提供 `APP_TOKEN`。
- 当 `APP_TOKEN` 缺失或仅包含空白字符时，启动立即失败。
- `just uv-run` 作为当前正式入口，默认直接绑定 `0.0.0.0`，便于本机、局域网和后续 VPS 调试时复用同一条启动路径。
- 绑定地址属于入口层语义，不写成应用级全局默认；因此其他未显式传 `--host` 的启动方式仍应自然保持 `127.0.0.1`。
- 认证文件路径通过显式环境变量指定，不把会话状态写死到不可控位置。

## 为什么现在要锁定 `mijiaAPI` 版本

当前登录初始化直接复用了 `mijiaAPI` 内部的扫码流程步骤，而不只是调用一个公开稳定的高级接口。这意味着：

- 版本漂移会直接影响登录初始化能否继续工作。
- 如果不锁版本，后续某次依赖升级可能在没有显式代码改动的情况下把登录链路悄悄打坏。
- 当前任务已经以本地环境中的 `mijiaAPI 3.0.5` 完成了导入与路由闭环验证，因此应该把这个版本冻结为当前代码基线。

因此，本任务把“当前 `uv.lock` 已解析并验证的 `mijiaAPI 3.0.5`”作为实现基线；后续若要升级版本，应单独开 task 验证扫码流程、设备发现与插座控制链路。

## 为什么需要专用登录页面

只提供 API 与文档页虽然能完成登录，但操作路径仍然偏底层：

- 操作者需要自己理解 `start -> 扫码 -> finish` 三步 API 顺序。
- 文档页更适合接口调试，不适合作为面向实际使用的扫码入口。
- 当项目部署到 VPS 后，专用登录页比 `curl` 或手动点文档接口更接近日常运维方式。

因此，本任务把 `/login` 作为第一阶段的专用登录页面落地。它不只负责扫码，还负责在登录完成后展示设备列表与 `power` 控制入口。页面本身只承担交互壳页职责，不放宽后端鉴权规则：页面内的所有实际请求仍然统一走 `Authorization: Bearer <TOKEN>`。

## 为什么 `/login` 页面还要提供 curl / JS 示例

第一阶段的推荐使用路径是：

- 先在 `/login` 页面里输入 `APP_TOKEN`。
- 在同一页面完成扫码登录。
- 在同一页面确认设备列表、属性选择与开关控制都已经跑通。
- 页面基于当前设备与当前属性选择结果，直接生成可复制的 curl 或 JS 示例代码。
- 用户复制自己需要的模式，把占位 token 替换成真实 `APP_TOKEN` 后，再放进 `Tasker`、脚本或其他调用端。

这样设计有两个直接收益：

- 页面先承担“把登录和设备控制联调跑通”的职责，用户不需要在还没确认 `did` 和 `property_name` 之前就手写请求。
- 示例代码由页面按当前设备上下文直接生成，用户只需要复制并替换占位 token，不需要自己再拼 URL、请求头和请求体。
- 默认先展示 curl，是为了让最直接的命令行调用路径保持最短；同时提供 JS，是为了让脚本接入端不必再把 curl 手工改写成 `fetch` 请求。

## 为什么扫码页不能把等待动作放到“扫码之后再点 finish”

`mijiaAPI` 原始二维码登录流程在拿到二维码后，会立即对 `lp` 发起最长 `120` 秒的长轮询请求，然后在这段时间里等待用户扫码确认。也就是说，底层语义更接近“先开始等待，再扫码”，而不是“扫完码后再补一个 finish 请求”。

如果页面把等待动作推迟到扫码完成之后才触发，就会偏离底层库已经验证过的控制流，进而放大“二维码很快失效”或“扫码后无法完成登录”的概率。

因此，本任务把 `/login` 页面流程冻结为：

- 点击“开始扫码登录”后生成二维码。
- 页面立即发起登录完成等待请求。
- 用户在这段等待窗口内使用米家 App 扫码确认。
- 成功后页面显示登录结果；若超时，再由用户手动重试等待或重新生成二维码。

## Docker 为什么放到后续任务

Docker 不只是启动命令，还涉及：

- 镜像构建 contract。
- 容器内依赖安装方式。
- 认证文件与状态目录的挂载语义。
- 公网 VPS 上的端口暴露、反向代理和重启策略。

这些都应在后续独立任务中冻结，而不是在当前最小 `uv run` 路线里顺手带过。
