# 公网安全论述

## 为什么 Spec 需要先定义安全边界

这个项目的目标部署环境是公网 VPS。只要服务最终会暴露到公网，安全性就不能被视为实现阶段的“附加优化”，而必须先进入 Spec。否则后续很容易出现一种常见错误：接口功能先跑通，认证、文档暴露、错误信息和部署收口在最后临时补丁式追加。

这种做法的问题在于，一旦运行时 contract 已经散落在代码里，再补安全约束往往会引入第二套入口、第二套鉴权或第二套配置语义，最终污染整体结构。

## Spec 在这个项目里如何保证安全性

当前 Spec 不是在“证明系统绝对安全”，而是在提前冻结几条不能后退的最小安全真值：

- 服务不得在缺失 `APP_TOKEN` 的情况下启动。
- 服务不得自动生成默认 token，避免出现“能跑但认证口径不明确”的状态。
- 健康检查只返回最小存活信息，不回显账号、设备或配置细节。
- 文档内容默认不公开；只有提供正确 token 后，OpenAPI schema 与交互式调试能力才会加载。
- 正式业务接口默认走 `Authorization: Bearer <TOKEN>`，不允许把 token 放进 query string。
- 只有 `just uv-run` 这个当前正式调试入口默认绑定 `0.0.0.0`，其他未显式传 `--host` 的启动入口都保持 `127.0.0.1`，避免把“为了测试方便”的绑定语义扩散成全局默认暴露面。
- 当前最小执行闭环只先跑本地 `uv run`，不在这一轮把公网 Docker 暴露细节草率写死。

这些约束的价值不在于它们覆盖了全部安全问题，而在于后续实现不能绕开这些底线。

## 当前阶段的安全边界

本任务已经落地第一阶段最小闭环应用，因此它能直接保证的是：

- 缺失或空白 `APP_TOKEN` 时，服务启动失败。
- 文档壳页不直接暴露 schema；只有输入 token 后才加载文档内容。
- `/login`、`/docs`、`/redoc` 作为壳页允许匿名打开，但页面内实际调用的接口与 `/openapi.json` 都遵循 Bearer token 鉴权。
- 除 `/healthz`、`/login`、`/docs`、`/redoc` 壳页外，其余请求默认需要 Bearer token。
- 只有 `just uv-run` 这个入口默认对外绑定 `0.0.0.0`，其余入口仍以 `127.0.0.1` 作为安全默认值。

还不能在本任务中完全解决的内容包括：

- HTTPS 终止与证书管理。
- 反向代理、限流和防火墙收口。
- 登录接口本身的暴露策略。
- 审计日志、访问日志脱敏与异常告警。

这些内容不应假装已经完成，而应在后续部署任务中继续冻结。
