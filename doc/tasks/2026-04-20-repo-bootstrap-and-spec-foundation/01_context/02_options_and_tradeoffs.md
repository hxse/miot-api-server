# 方案对比与取舍

## 方案 1：复用 `Do1e/mijia-api`

优点：

- 已经具备二维码登录、认证数据持久化、设备列表获取与属性写入能力。
- 代码体量更小，更接近当前项目“薄封装 `FastAPI` 服务”的定位。
- 能直接承接“首次扫码登录 -> 后续复用认证文件 -> 拉设备 -> 控制设备”的主链。

问题：

- 不是官方 SDK，登录流程与内部实现存在版本波动风险。
- 需要我们自己定义服务端 API、鉴权、页面与部署方式。

## 方案 2：手动解析 `XiaoMi/ha_xiaomi_home`

优点：

- 官方支持，设备导入、OAuth、MIoT-Spec 处理与长期集成策略更完整。
- 对“帐号登录 -> 设备选择”的体验设计有参考价值。

问题：

- 代码强耦合 Home Assistant 宿主，拆出最小子集的成本很高。
- 对当前只做轻量远程控制服务的目标来说，明显过重。

## 技术路线结论

当前主路线选择复用 `Do1e/mijia-api`。`XiaoMi/ha_xiaomi_home` 只作为参考资料，用来校对登录链路、设备选择体验与 MIoT 模型，不作为主要实现底座。

## 方案 3：设备作为一级资源，能力作为子路由

候选形式：

- `GET /devices`
- `GET /devices/{did}`
- `POST /devices/{did}/power/on`
- `POST /devices/{did}/power/off`

优点：

- 路由天然表达“这是某个设备上的某个能力”。
- 与米家设备列表的认知模型一致。
- 后续扩展能力时可以继续沿用 `devices/{did}/<capability>/...` 的结构。

问题：

- 设备能力描述、页面结构与错误语义都需要一起设计清楚。
- 如果某设备存在多个候选属性，调用方必须有明确选择路径。

## 方案 4：通用能力 DSL 或万能控制入口

候选形式：

- `POST /devices/{did}/properties/set`
- `POST /actions/invoke`
- 通用 JSON DSL

优点：

- 理论上扩展性更强。
- 能更直接暴露底层 MIoT 能力。

问题：

- 第一阶段范围明显过大。
- 会把大量底层协议细节、属性命名与失败语义暴露给调用方。
- 对当前只先正式支持 `power` 能力的目标来说，复杂度不成比例。

## 控制模型结论

当前正式选择方案 3。第一阶段对外只正式支持 `power` 能力，不引入通用能力 DSL。

## 方案 5：同步 provider 业务路由继续使用 `async def`

优点：

- 路由声明表面上统一为异步形态。

问题：

- 当前 `MijiaProvider` 和底层 `mijiaAPI` 仍然是同步实现。
- 在 `async def` 路由里直接调用同步 provider，会直接阻塞事件循环。
- 只是把同步函数包进 `async` 语法，并不会把同步 HTTP 自动变成真正异步。

## 方案 6：同步 provider 业务路由统一使用 `def`

优点：

- 与当前依赖真实语义一致。
- FastAPI 会在线程池里执行这些路由，避免在事件循环里直接承接同步云请求。
- 不改变公开 HTTP contract。

问题：

- 单次请求仍可能较慢；只是阻塞归属从事件循环线程转移到工作线程。

## 执行模型结论

当前正式选择方案 6。只要 provider 仍然是同步实现，直接调用 provider 的业务路由就统一使用 `def`。
