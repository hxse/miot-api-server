# 问题背景

`todo.md` 的原始目标很集中：在公网 VPS 上部署一个轻量 `FastAPI` 服务，通过小米云远程控制家里的米家设备，并让安卓手机上的 `Tasker` 通过 HTTP 请求触发动作。

这条链路技术上可行，原因没有变：

- 控制路径建立在小米云控上，不依赖设备与 VPS 同网段。
- `mijia-api` 已经具备二维码登录、认证持久化、设备枚举与属性写入能力。
- 第一阶段只要把“登录 -> 看到设备 -> 触发控制”跑通，就能形成闭环。

真正困难的地方不在“能不能做”，而在“第一阶段最终真值应该如何收口”。当前这轮收口关注四个问题：

- 对外接口应该直接围绕设备与能力表达，而不是引入额外的抽象资源。
- 登录、设备发现、设备控制、测试重置与文档访问需要共享统一的鉴权口径与运行时约束。
- 当前底层依赖仍然是同步实现，路由执行模型必须与真实阻塞语义保持一致。
- 仓内只保留一份第一阶段 task，避免真值分散。

因此当前 task 的目标不是记录演化历史，而是把今天这版正式设计收口成一套可直接执行、可直接验收的单一真值。

# 主要风险

- `mijia-api` 属于非官方依赖，版本波动、登录流程变化和上游接口变更仍然是主要外部风险。
- 公网暴露 HTTP 接口必须从第一天就写清最小安全口径，尤其是 `APP_TOKEN`、文档加载策略与最小暴露面。
- 当前同步云控链路仍包含同步 HTTP、同步设备列表获取与显式 `time.sleep(...)`，这会影响请求阻塞归属和慢请求体验。
- 若要求用户手工填写 `siid/piid` 或暴露底层 MIoT 细节，会把内部协议解析成本转嫁给部署者。
