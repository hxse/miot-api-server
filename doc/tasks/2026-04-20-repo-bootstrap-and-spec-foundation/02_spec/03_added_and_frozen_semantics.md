# 新增语义逻辑与冻结语义逻辑

## Added Semantics

- 新增“运行时真值下沉到 task”的项目约束，避免把阶段性业务细节写进 `AGENTS.md`。
- 新增第一阶段必须包含登录初始化与设备发现能力的正式判断。
- 新增“当前任务只先打通 `uv run`”的执行约束。
- 新增 `APP_TOKEN` 显式传入、禁止自动生成、空白即失败的正式约束。
- 新增公网 VPS 场景下的最小安全 contract，包括 token 保护的文档内容加载与最小暴露面要求。
- 新增第一阶段最小闭环对应的业务实现：登录初始化、设备发现、设备能力描述、设备级 `power` 控制、`/auth/reset`、专用登录页与健康检查。
- 新增当前 `uv.lock` 对应 `mijiaAPI 3.0.5` 的验证基线，用于约束登录链路依赖的私有实现细节。
- 新增专用登录页面 `/login`，用于承载扫码展示、登录完成交互、设备能力展示、页面内控制入口与可复制的 curl / JS 示例代码，而不是只依赖 `curl` 或文档页。
- 新增 `/login` 的自动等待语义：二维码生成后立即开始等待扫码确认，贴近 `mijiaAPI` 原始长轮询流程。
- 新增服务端基于 `login_url` 生成本地二维码数据的约束，避免前端直接依赖第三方图链。
- 新增“无需全局绑定”的设备能力导向控制模型。
- 新增“同步 provider 业务路由统一使用 `def`”的运行时执行语义。

## Frozen Semantics

- 项目定位冻结为“基于 `mijia-api` 的薄 `FastAPI` 封装”。
- 当前主要技术路线冻结为复用 `Do1e/mijia-api`，而不是手动拆解 `XiaoMi/ha_xiaomi_home` 作为主实现底座。
- 第一阶段不把用户手工填写 `siid/piid` 作为正式部署前置条件。
- 第一阶段只正式支持 `power` 能力，不扩展通用能力 DSL。
- `Authorization: Bearer <TOKEN>` 仍是所有业务请求的正式认证口径。
- 本任务不产出 Dockerfile、镜像或公网部署编排。
