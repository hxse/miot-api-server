# 审查问题与取舍

## 为什么必须收紧 power 识别

当前 Provider 将所有可写布尔属性纳入 `power` 候选，并在只有一个候选时自动作为默认属性使用。这个做法实现简单，但存在 quietly wrong 风险：某些设备只有一个可写布尔属性时，该属性不一定表达电源开关。

本任务选择收紧识别规则，而不是继续依赖“单候选自动可信”：

- `power` 路由是对外控制入口，错误执行的成本高于漏识别。
- 第一阶段目标是安全地支持明确 power 能力，不是最大化覆盖所有布尔属性。
- 遇到无法确认的布尔属性时，返回“不支持 power”比误执行更符合当前安全基线。

## 为什么不引入手工 siid/piid 回退

一个可选方案是允许用户手工传入 `siid/piid` 或任意属性名，绕过 Provider 的能力识别限制。该方案会把 MIoT 内部细节重新暴露给部署者，背离第一阶段“无需手工填写 `siid/piid`”的冻结语义。

因此，本任务只收紧 Provider 内部识别规则，不新增通用属性写入口，也不新增兼容回退。

## Docker 与仓库入口的处理方式

Docker 旧 task 的执行记录反映当时已经完成的事实，不作为长期全局最新验收记录回写。本任务把补充验证结果记录在新的 `04_review` 中，避免历史 task 反复回改。

`just check` 与 `just format` 已经出现在当前工作副本。与其把它们视为临时入口，本任务将其正式纳入仓库入口 contract：`check` 负责类型检查，`format` 负责格式化代码。
