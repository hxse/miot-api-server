# 输入边界取舍

## 为什么只收紧请求体

当前服务的主要风险边界有两类：

- 公网调用方提交的请求体。
- 米家上游返回的设备与 spec 数据。

公网请求体应该尽早失败，因为它直接触发登录等待或设备控制。尤其是空字符串、全空白字符串和异常超长字符串，越早在 schema 层拦截，错误语义越清晰。

米家上游数据不适合用全局 strict response model 处理。上游字段类型可能存在小幅漂移，Provider 层已经负责必要归一化；若响应模型全局 strict，容易把上游小漂移变成不稳定的 500。

## 为什么 timeout_seconds 不改成 strict int

`timeout_seconds` 当前已有 `ge=1` 与 `le=180` 约束。Tasker、curl 或简单脚本调用时，数字字符串 `"120"` 很常见。当前阶段没有必要把这类输入拒绝为 422。

因此，本任务只保持范围校验，不开启 strict int。

## 为什么不引入 pydantic-settings

当前配置层只负责 `APP_TOKEN`、认证文件路径和 spec 缓存路径。现有实现已经清晰表达“缺失 token 启动失败”和“路径不能为空”的失败语义。引入 `pydantic-settings` 会增加抽象层，但不会明显改善当前问题。
